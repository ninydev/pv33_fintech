# Концепция P2P-сети для dApp

Этот документ описывает простую архитектуру для объединения нескольких нод нашего блокчейна в единую децентрализованную сеть (Peer-to-Peer).

## Основная идея

Каждая нода в сети является одновременно и клиентом, и сервером. Она хранит полную копию блокчейна и обменивается обновлениями с другими участниками сети (пирами).

## Технологический стек

1.  **HTTP Server (`Express.js`):** Предоставляет REST API для управления нодой.
2.  **WebSockets (`ws`):** Используется для мгновенного обмена данными между нодами в реальном времени.

## Архитектура взаимодействия

### 1. HTTP API (Интерфейс управления)

Каждая нода запускает HTTP-сервер, который слушает определенный порт и предоставляет следующие эндпоинты:

-   `GET /blocks`
    -   **Назначение:** Получить текущее состояние блокчейна на данной ноде.
    -   **Ответ:** JSON-массив со всеми блоками в цепочке.

-   `POST /mineBlock`
    -   **Назначение:** Создать (добыть) новый блок. В теле запроса передаются данные для нового блока.
    -   **Действие:** Нода создает новый блок, добавляет его в свою цепочку и затем **транслирует** свою обновленную цепочку всем пирам через WebSockets.

-   `GET /peers`
    -   **Назначение:** Посмотреть список пиров, к которым подключена нода.
    -   **Ответ:** Массив адресов пиров (например, `["ws://localhost:6002", "ws://localhost:6003"]`).

-   `POST /addPeer`
    -   **Назначение:** Установить соединение с новой нодой в сети. В теле запроса передается WebSocket-адрес пира.
    -   **Действие:** Нода инициирует WebSocket-соединение с указанным пиром.

### 2. WebSocket (Синхронизация)

WebSocket-соединения являются "кровеносной системой" нашей сети.

-   **Подключение:** При запуске нода может автоматически подключаться к списку известных ей пиров. Новые пиры добавляются через эндпоинт `/addPeer`.

-   **Обмен данными:** Основной тип сообщений, передаваемых по WebSocket — это вся цепочка блоков.

-   **Механизм консенсуса (самый простой): "Побеждает самая длинная цепь"**
    1.  Когда нода `А` майнит новый блок, она рассылает свою (теперь уже более длинную) цепочку всем своим пирам (нодам `Б` и `В`).
    2.  Нода `Б` получает цепочку от ноды `А`.
    3.  Она проверяет её валидность (`isChainValid()`).
    4.  Она сравнивает длину полученной цепи со своей.
    5.  Если полученная цепь **валидна и длиннее** её собственной, нода `Б` **заменяет** свою цепочку на полученную.
    6.  Если нет — игнорирует её.

Этот механизм гарантирует, что со временем все ноды в сети придут к единому состоянию (консенсусу).

## Пример запуска сети

1.  Запускаем Ноду 1 на порту `3001` (HTTP) и `6001` (WebSocket).
2.  Запускаем Ноду 2 на порту `3002` (HTTP) и `6002` (WebSocket).
3.  Отправляем POST-запрос на `http://localhost:3001/addPeer` с телом `{"peer": "ws://localhost:6002"}`. Теперь ноды соединены.
4.  Отправляем POST-запрос на `http://localhost:3002/mineBlock` с какими-то данными.
5.  Нода 2 создает блок, её цепь становится длиннее. Она тут же отправляет свою новую цепь Ноде 1 по WebSocket.
6.  Нода 1 видит, что полученная цепь длиннее и валидна, и обновляет свою. Теперь обе ноды снова синхронизированы.
